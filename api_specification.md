# API Endpoint Specification (Instagram-like System)

This document defines the comprehensive, production-grade API specification for our Instagram-like social media platform. It covers RESTful and GraphQL endpoints, authentication mechanisms, media upload workflows, cursor-based pagination, rate limiting, and error handling.

---

## 1. Architectural Overview & Protocols

To support high-throughput, low-latency operations, and a rich client experience, the platform utilizes a hybrid API architecture:

1. **RESTful APIs**: Used for resource-oriented, deterministic actions such as authentication, post creation, media upload coordination, and state-changing operations (likes, follows, comments).
2. **GraphQL APIs**: Used for flexible, client-driven data fetching, such as rendering complex user profiles, home feeds, and explore feeds where clients need to request specific fields to minimize over-fetching and round-trips.
3. **WebSockets**: Used for real-time notifications, direct messaging, and live activity feeds.

### 1.1 Global Headers

All REST requests must include the following headers:

| Header | Type | Description | Example |
| :--- | :--- | :--- | :--- |
| `Content-Type` | String | Media type of the request body. Required for POST/PUT/PATCH. | `application/json` |
| `Accept` | String | Expected response media type. | `application/json` |
| `Authorization` | String | Bearer token for authenticated endpoints. | `Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...` |
| `X-Client-Version` | String | Client application version for tracking and deprecation. | `ios-v4.12.0` |
| `X-Correlation-ID` | String | Unique UUID generated by the client for distributed tracing. | `f81d4fae-7dec-11d0-a765-00a0c91e6bf6` |

### 1.2 Rate Limiting

To prevent abuse and ensure high availability, rate limiting is enforced at the API Gateway layer using a token bucket algorithm.

**Rate Limit Headers returned in every response:**
* `X-RateLimit-Limit`: The maximum number of requests allowed in the current window (e.g., `100`).
* `X-RateLimit-Remaining`: The number of remaining requests allowed in the current window (e.g., `85`).
* `X-RateLimit-Reset`: The Unix epoch timestamp indicating when the current rate limit window resets (e.g., `1672531200`).

**Default Limits (IP and User-based):**
* **Standard Authenticated Endpoints**: 100 requests per minute per user.
* **Feed Retrieval Endpoints**: 60 requests per minute per user.
* **Authentication Endpoints (Login/Signup)**: 10 requests per minute per IP.
* **Media Upload Initiation**: 20 requests per minute per user.

When a limit is exceeded, the server returns a `429 Too Many Requests` status code with a JSON payload indicating the retry delay.

## 2. Authentication & Session Management

We implement a stateless, secure JWT-based authentication flow with **Access Tokens** (short-lived, 15 minutes) and **Refresh Tokens** (long-lived, 7 days, stored in secure, HttpOnly, SameSite=Strict cookies, or rotated via API).

### 2.1 Sign Up
* **Endpoint**: `POST /api/v1/auth/signup`
* **Authentication**: None
* **Description**: Registers a new user account.

#### Request Body
```json
{
  "email": "user@example.com",
  "username": "johndoe",
  "password": "SecurePassword123!",
  "full_name": "John Doe"
}
```

#### Responses
* **`201 Created`**: Account successfully created.
  ```json
  {
    "status": "success",
    "message": "User registered successfully. Please verify your email.",
    "data": {
      "user": {
        "id": "usr_9a8b7c6d5e4f",
        "email": "user@example.com",
        "username": "johndoe",
        "full_name": "John Doe",
        "is_verified": false,
        "created_at": "2023-10-27T10:14:00Z"
      }
    }
  }
  ```
* **`400 Bad Request`**: Validation failed (e.g., weak password, invalid email format).
  ```json
  {
    "status": "error",
    "code": "VALIDATION_FAILED",
    "message": "Validation failed for the input parameters.",
    "details": [
      {
        "field": "password",
        "issue": "Password must be at least 8 characters long and contain at least one uppercase letter, one number, and one special character."
      }
    ]
  }
  ```
* **`409 Conflict`**: Username or email already exists.
  ```json
  {
    "status": "error",
    "code": "RESOURCE_ALREADY_EXISTS",
    "message": "A user with this email or username already exists."
  }
  ```

### 2.2 Login
* **Endpoint**: `POST /api/v1/auth/login`
* **Authentication**: None
* **Description**: Authenticates a user and returns access and refresh tokens.

#### Request Body
```json
{
  "username_or_email": "johndoe",
  "password": "SecurePassword123!"
}
```

#### Responses
* **`200 OK`**: Authentication successful.
  ```json
  {
    "status": "success",
    "data": {
      "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
      "token_type": "Bearer",
      "expires_in": 900,
      "refresh_token": "ref_8f7e6d5c4b3a21...",
      "user": {
        "id": "usr_9a8b7c6d5e4f",
        "username": "johndoe",
        "email": "user@example.com",
        "full_name": "John Doe",
        "profile_picture_url": "https://cdn.instagram-clone.com/profiles/usr_9a8b7c6d5e4f.jpg",
        "is_verified": true
      }
    }
  }
  ```
* **`401 Unauthorized`**: Invalid credentials.
  ```json
  {
    "status": "error",
    "code": "INVALID_CREDENTIALS",
    "message": "The username or password you entered is incorrect."
  }
  ```

### 2.3 Token Refresh
* **Endpoint**: `POST /api/v1/auth/refresh`
* **Authentication**: None (Requires Refresh Token in body or HttpOnly cookie)
* **Description**: Rotates the refresh token and issues a new short-lived access token to prevent session hijacking.

#### Request Body
```json
{
  "refresh_token": "ref_8f7e6d5c4b3a21..."
}
```

#### Responses
* **`200 OK`**: Tokens rotated successfully.
  ```json
  {
    "status": "success",
    "data": {
      "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9_new...",
      "token_type": "Bearer",
      "expires_in": 900,
      "refresh_token": "ref_new_9f8e7d6c5b4a..."
    }
  }
  ```
* **`401 Unauthorized`**: Invalid or expired refresh token.
  ```json
  {
    "status": "error",
    "code": "INVALID_REFRESH_TOKEN",
    "message": "The refresh token is invalid or has expired. Please log in again."
  }
  ```

---

### 2.4 Logout
* **Endpoint**: `POST /api/v1/auth/logout`
* **Authentication**: Bearer Token
* **Description**: Invalidates the current refresh token and blacklists the access token until its natural expiration.

#### Request Body
```json
{
  "refresh_token": "ref_8f7e6d5c4b3a21..."
}
```

#### Responses
* **`200 OK`**: Logged out successfully.
  ```json
  {
    "status": "success",
    "message": "Successfully logged out and session invalidated."
  }
  ```

## 3. Post Creation & Media Upload Flow

To optimize performance and minimize server load, media uploads do not pass directly through the application servers. Instead, we use a **Pre-signed URL Upload Flow**:

```
[Client] ---> 1. POST /posts/upload-url ---> [App Server]
   ^                                              |
   | <------- 2. Returns Pre-signed S3 URL -------|
   |
   |--------> 3. PUT Binary Media File ---------> [S3 Bucket]
   |                                              |
   | <------- 4. 200 OK (Upload Complete) --------|
   |
   |--------> 5. POST /posts (with media key) --> [App Server]
   |                                              |
   | <------- 6. 201 Created (Post Live) ---------|
```

### 3.1 Request Pre-signed Upload URL
* **Endpoint**: `POST /api/v1/posts/upload-url`
* **Authentication**: Bearer Token
* **Description**: Requests a secure, short-lived pre-signed URL to upload media directly to an S3 bucket.

#### Request Body
```json
{
  "file_name": "vacation_photo.jpg",
  "content_type": "image/jpeg",
  "file_size_bytes": 4194304
}
```

#### Responses
* **`200 OK`**: Pre-signed URL generated.
  ```json
  {
    "status": "success",
    "data": {
      "upload_url": "https://s3.amazonaws.com/instagram-clone-media/uploads/usr_9a8b/vacation_photo.jpg?AWSAccessKeyId=AKIAIOSFODNN7EXAMPLE&Signature=vjbyPxybdZaNmGa%2ByT272YEAiv4%3D&Expires=1672531800",
      "media_key": "uploads/usr_9a8b/vacation_photo.jpg",
      "expires_at": "2023-10-27T10:30:00Z"
    }
  }
  ```
* **`400 Bad Request`**: Unsupported file type or file size exceeds maximum limit (e.g., > 10MB for images, > 100MB for videos).

### 3.3 Edit Post
* **Endpoint**: `PATCH /api/v1/posts/{post_id}`
* **Authentication**: Bearer Token
* **Description**: Updates the caption, location, or tagged users of an existing post. Only the post owner can perform this action.

#### Request Body
```json
{
  "caption": "Chasing sunsets in Bali! 🌅 (Updated caption)",
  "location": null,
  "tagged_user_ids": ["usr_1a2b3c4d5e6f"]
}
```

#### Responses
* **`200 OK`**: Post updated successfully.
* **`403 Forbidden`**: User is not the owner of the post.
* **`404 Not Found`**: Post does not exist.

---

### 3.4 Delete Post
* **Endpoint**: `DELETE /api/v1/posts/{post_id}`
* **Authentication**: Bearer Token
* **Description**: Deletes a post and triggers background jobs to clean up media files from S3 and remove associated likes, comments, and hashtags.

#### Responses
* **`200 OK`**: Post deleted successfully.
  ```json
  {
    "status": "success",
    "message": "Post was successfully deleted."
  }
  ```
* **`403 Forbidden`**: User is not authorized to delete this post.

## 4. Feed Retrieval & Pagination

To handle high-throughput feeds efficiently, we use **Cursor-Based Pagination** (using a base64-encoded cursor containing the timestamp and ID of the last item) instead of offset-based pagination. This avoids performance degradation and duplicate/skipped items when new posts are created.

### 4.1 Home Feed (REST Version)
* **Endpoint**: `GET /api/v1/feed/home`
* **Authentication**: Bearer Token
* **Description**: Retrieves the personalized home feed for the authenticated user, containing posts from followed users.

#### Query Parameters
* `limit` (integer, optional, default: `20`, max: `50`): Number of posts to return.
* `cursor` (string, optional): Base64 encoded cursor for pagination.

#### Responses
* **`200 OK`**: Feed retrieved successfully.
  ```json
  {
    "status": "success",
    "data": {
      "posts": [
        {
          "id": "pst_3c4d5e6f7g8h",
          "user": {
            "id": "usr_9a8b7c6d5e4f",
            "username": "johndoe",
            "profile_picture_url": "https://cdn.instagram-clone.com/profiles/usr_9a8b7c6d5e4f.jpg"
          },
          "media_url": "https://cdn.instagram-clone.com/media/posts/pst_3c4d5e6f7g8h.jpg",
          "caption": "Chasing sunsets in Bali! 🌅",
          "likes_count": 1240,
          "comments_count": 42,
          "has_liked": true,
          "created_at": "2023-10-27T10:35:00Z"
        }
      ],
      "pagination": {
        "next_cursor": "eyJjcmVhdGVkX2F0IjoiMjAyMy0xMC0yN1QxMDozNTowMFoiLCJpZCI6InBzdF8zYzRkNWU2ZjdnOGgifQ==",
        "has_more": true
      }
    }
  }
  ```

---

### 4.2 Explore Feed (REST Version)
* **Endpoint**: `GET /api/v1/feed/explore`
* **Authentication**: Bearer Token (Optional)
* **Description**: Retrieves trending or algorithmically recommended posts.

#### Query Parameters
* `limit` (integer, optional, default: `24`): Number of items.
* `cursor` (string, optional): Pagination cursor.
* `category` (string, optional): Filter by category (e.g., `travel`, `food`, `fashion`).

## 7. GraphQL Schema Definition

For complex, nested data fetching (e.g., rendering a rich user profile with their recent posts, follower counts, and mutual friends in a single network request), we expose a GraphQL endpoint at `/api/graphql`.

### 7.1 Schema Definition (SDL)

```graphql
type User {
  id: ID!
  username: String!
  fullName: String!
  bio: String
  website: String
  profilePictureUrl: String
  isPrivate: Boolean!
  isVerified: Boolean!
  followersCount: Int!
  followingCount: Int!
  postsCount: Int!
  relationship: RelationshipStatus!
  posts(limit: Int, cursor: String): PostConnection!
}

type RelationshipStatus {
  isFollowing: Boolean!
  isFollowedBy: Boolean!
  isBlocked: Boolean!
  hasPendingRequest: Boolean!
}

type Post {
  id: ID!
  user: User!
  mediaUrl: String!
  caption: String
  location: Location
  likesCount: Int!
  commentsCount: Int!
  hasLiked: Boolean!
  taggedUsers: [User!]!
  comments(limit: Int, cursor: String): CommentConnection!
  createdAt: String!
}

type Location {
  name: String!
  latitude: Float
  longitude: Float
}

type PostConnection {
  edges: [PostEdge!]!
  pageInfo: PageInfo!
}

type PostEdge {
  node: Post!
  cursor: String!
}

type Comment {
  id: ID!
  post: Post!
  user: User!
  content: String!
  createdAt: String!
  replies(limit: Int, cursor: String): CommentConnection!
}

type CommentConnection {
  edges: [CommentEdge!]!
  pageInfo: PageInfo!
}

type CommentEdge {
  node: Comment!
  cursor: String!
}

type PageInfo {
  endCursor: String
  hasNextPage: Boolean!
}

type Query {
  me: User
  user(username: String!): User
  post(id: ID!): Post
  homeFeed(limit: Int, cursor: String): PostConnection!
  exploreFeed(limit: Int, cursor: String, category: String): PostConnection!
}

type Mutation {
  likePost(postId: ID!): Post!
  unlikePost(postId: ID!): Post!
  addComment(postId: ID!, content: String!, parentCommentId: ID): Comment!
  deleteComment(commentId: ID!): Boolean!
}
```

### 7.2 Example GraphQL Query
To render a user's profile screen with their first 3 posts:

```graphql
query GetUserProfile($username: String!) {
  user(username: $username) {
    username
    fullName
    bio
    followersCount
    followingCount
    postsCount
    relationship {
      isFollowing
    }
    posts(limit: 3) {
      edges {
        node {
          id
          mediaUrl
          likesCount
          commentsCount
        }
      }
      pageInfo {
        endCursor
        hasNextPage
      }
    }
  }
}
```

## 8. Global Error Codes & Handling

All error responses follow a standardized JSON structure to allow clients to handle errors programmatically.

### 8.1 Error Response Schema
```json
{
  "status": "error",
  "code": "ERROR_CODE_STRING",
  "message": "A human-readable explanation of the error.",
  "details": [] // Optional array of specific field validation errors
}
```

### 8.2 Standard Error Codes

| HTTP Status | Error Code | Description |
| :--- | :--- | :--- |
| `400` | `VALIDATION_FAILED` | The request parameters or body failed validation. |
| `401` | `UNAUTHORIZED` | Authentication token is missing, invalid, or expired. |
| `403` | `FORBIDDEN` | The authenticated user does not have permission to perform this action. |
| `404` | `RESOURCE_NOT_FOUND` | The requested resource (user, post, comment) does not exist. |
| `409` | `RESOURCE_ALREADY_EXISTS` | Conflict with an existing resource (e.g., username already taken). |
| `422` | `UNPROCESSABLE_ENTITY` | Semantic errors in the request body (e.g., uploading a corrupted file). |
| `429` | `RATE_LIMIT_EXCEEDED` | Too many requests have been sent in a given time frame. |
| `500` | `INTERNAL_SERVER_ERROR` | An unexpected error occurred on the server. |
| `503` | `SERVICE_UNAVAILABLE` | The server is temporarily unable to handle the request (e.g., database maintenance). |
